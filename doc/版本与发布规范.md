# MindTrain 版本与发布规范

本文约定 MindTrain 软件版本、镜像标签和正式 Release 的统一维护方式。题目版本、数据库迁移版本和 MCP 契约版本不属于软件版本。

## 1. 版本格式

正式版本统一使用 [Semantic Versioning](https://semver.org/)：`MAJOR.MINOR.PATCH`，例如 `1.0.0`。

- `MAJOR`：存在不兼容的产品、API 或数据行为变更。
- `MINOR`：向后兼容的新功能。
- `PATCH`：向后兼容的问题修复。
- Maven 开发版本使用 `<下一版本>-SNAPSHOT`。
- Codex Plugin 开发构建可使用 `<基础版本>+codex.<UTC时间戳>`；正式 Release 使用纯版本号。

Git Tag 和 GitHub Release 在软件版本前加 `v`，例如 `v1.0.0`；代码和制品内部仍使用 `1.0.0`。

## 2. 需要统一维护的位置

一次正式发布中的 Core、MCP、Web、Plugin 和 OpenAPI 使用同一个软件版本。

| 位置 | Release 示例 | 维护要求 |
| --- | --- | --- |
| 根 `pom.xml` | `1.0.0` | Core 与 MCP 的基础版本 |
| `apps/trainer-core/pom.xml` 父版本 | `1.0.0` | 与根 Maven 版本一致 |
| `apps/trainer-mcp/pom.xml` 父版本 | `1.0.0` | 与根 Maven 版本一致 |
| `apps/web/package.json` | `1.0.0` | 同步更新 `pnpm-lock.yaml` |
| `plugins/mindtrain/.codex-plugin/plugin.json` | `1.0.0` | 正式 Plugin 版本 |
| `contracts/openapi/trainer-core.yaml` | `1.0.0` | OpenAPI `info.version` |
| 版本相关测试断言 | `1.0.0` | 与实际服务端和 Plugin 规则一致 |

`McpCompatibility.MINIMUM_PLUGIN_VERSION` 只在旧 Plugin 与新服务端确实不兼容时提高，不因每次发布自动跟随软件版本。

## 3. Docker、Git 与 Release 标签

| 类型 | 规则 |
| --- | --- |
| Docker 精确版本 | `1.0.0`，内容发布后永不移动 |
| Docker 最新正式版 | `latest`，只在正式 Release 时移动 |
| Docker 开发版 | `dev`，由任意短期功能分支手动发布，允许移动 |
| Docker 提交版本 | `sha-<commit>`，用于定位和回滚 |
| Git Tag | `v1.0.0`，指向已验证的 Release 提交，禁止重写 |
| GitHub Release | 与 Git Tag 同名，例如 `v1.0.0` |

Core、MCP 和 Web 三个正式镜像只发布 `1.0.0` 与 `latest`，不额外维护 `1.0` 浮动标签。生产部署优先固定精确版本；只有明确接受自动升级时才使用 `latest`。

CI 只在推送正式 Tag `vX.Y.Z` 时更新 `X.Y.Z` 与 `latest`；普通 `main` 提交只执行验证和镜像试构建，不覆盖正式镜像。

## 4. 不随软件版本同步的编号

| 编号 | 规则 |
| --- | --- |
| MCP 契约版本 | 只有协议发生破坏性变更时递增 |
| Flyway migration | 只追加新迁移，禁止修改或重排已发布迁移 |
| Question `version` | 仅表示单道题目的内容修订 |
| Java、Spring Boot、pnpm 等版本 | 按依赖升级独立维护 |
| 概要设计文档版本 | 表示文档修订，不等同软件 Release |

## 5. 正式发布清单

1. 从最新 `main` 创建 `codex/release-vX.Y.Z`。
2. 将第 2 节中的软件版本统一为 `X.Y.Z`，仅在不兼容时调整最低 Plugin 或契约版本。
3. 更新升级说明和 Release Notes，明确数据库迁移、兼容性和回滚方式。
4. 运行 `mvn verify`、`make check`、Web 测试与生产构建。
5. 通过 PR 合并到 `main`，确保 CI 全部通过。
6. 在合并提交创建并推送不可变 Tag `vX.Y.Z`，创建同名 GitHub Release。
7. 从该 Tag 构建 Core、MCP、Web，推送 `X.Y.Z` 与 `latest` 镜像。
8. 发布同版本 Codex Plugin，并验证 Plugin、MCP 和契约兼容性检查。
9. 备份数据库后升级实例，确认 Core、MCP、Web 均为 `healthy`，完成一次真实训练验证。

发布失败时不移动已有 Tag 和精确镜像标签；修复后发布新的 PATCH 版本。
