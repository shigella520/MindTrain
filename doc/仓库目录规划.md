# MindTrain 仓库目录规划

## 1. 目标

MindTrain 使用单仓库管理 Training Core、Trainer MCP、Web、Codex Plugin、契约和部署配置。生产题库、Prompt、Session、Attempt、掌握度和调度状态只保存在 Training Core 的 PostgreSQL 中，不再以 JSON/JSONL 文件维护在仓库内。

仓库保持模块化单体边界，不为尚未落地的能力预先创建空目录。

## 2. 当前目录

```text
MindTrain/
├── apps/
│   ├── trainer-core/       # 权威业务数据、判分、调度、报告和 REST API
│   ├── trainer-mcp/        # MCP Gateway，只调用 Core API
│   └── web/                # Dashboard、Web 训练和管理页
├── plugins/
│   └── mindtrain/          # Codex Plugin、唯一 Skill 源和本地 MCP Bridge
├── contracts/
│   ├── openapi/            # Training Core REST 契约
│   └── scheduler-spi/      # 调度 Provider 能力契约
├── deploy/
│   └── core-only/          # Core、MCP、Web、PostgreSQL Compose
├── assets/
│   └── brand/              # README 和产品使用的公共品牌素材
├── doc/                    # 产品、架构、部署和开发文档
├── scripts/                # 仓库级静态检查
├── tests/                  # Plugin 与仓库级测试
├── .agents/plugins/        # Codex Marketplace 清单
├── .github/workflows/      # CI 与镜像发布
├── Makefile
└── pom.xml
```

Java 和 Web 的模块测试分别放在对应应用目录中：

```text
apps/trainer-core/src/test/
apps/trainer-mcp/src/test/
apps/web/tests/
```

## 3. 依赖方向

```text
Codex Plugin ──> Trainer MCP ──> Training Core ──> PostgreSQL
MindTrain Web ─────────────────> Training Core
Future Anki Bridge ────────────> Training Core / Scheduler Contract
```

- 只有 Training Core 可以直接访问业务数据库。
- Trainer MCP、Web 和未来 Anki Bridge 只能通过 Core API 获取权威数据。
- Codex Skill 只编排 MCP 工具，不保存业务数据。
- Plugin Bridge 只保存私有实例 URL 和 Token，配置位于用户目录而不是仓库。
- Scheduler 实现必须位于 Provider 合同之后。

## 4. Skill 与 Plugin

`plugins/mindtrain/skills/mindtrain/` 是 MindTrain Skill 的唯一源码和发布位置。

不再维护仓库级第二份 Skill，也不保留早期 `java-interview-coach` 文件式训练原型。Marketplace 安装时直接加载 Plugin 中的 Skill、Bridge、图标和 MCP manifest。

Plugin 目录必须自包含：

```text
plugins/mindtrain/
├── .codex-plugin/plugin.json
├── .mcp.json
├── assets/
├── scripts/
└── skills/mindtrain/
```

## 5. 数据边界

仓库允许保存：

- Flyway migrations。
- OpenAPI 和 Scheduler 契约。
- 不含私人信息的测试代码内 Fixture。
- Docker Compose、环境变量样例和运维文档。
- 品牌图片和 README 展示素材。

仓库禁止保存：

- 用户题库和私人候选题。
- Session、Attempt、Mistake、Mastery 或 Review State。
- 真实 Token、密码、模型密钥和 Plugin 私有配置。
- 从生产数据库导出的数据文件。

历史 Flyway migration 不得因为功能退役而重写。旧表需要停用时，通过新的 migration 明确处理；包含历史审计价值的数据默认保留。

## 6. 测试边界

- Core 业务 Fixture 由测试代码写入测试数据库，不通过生产导入 API 造数。
- Plugin 测试验证 Marketplace、manifest、Bridge、安全配置和 Skill 元数据。
- Web 测试验证状态映射、配置和 Dashboard 计算。
- PR 必须执行 Maven、Plugin、Web 和 Docker 构建检查。

## 7. 后续扩展

只有在功能开始实现时才新增目录：

- Anki 集成落地时创建 `integrations/anki/`。
- 训练领域导入导出契约落地时创建 `contracts/domain-transfer/`。
- AI Eval Runner 落地时创建对应测试目录和最小 Eval Fixture。

新目录必须同时具有实现、测试或明确的构建入口，避免再次保留只有占位文件的规划目录。
