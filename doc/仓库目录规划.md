# MindTrain 仓库目录规划

## 1. 目的

本文把《概要设计》中的目标架构映射为可逐步落地的单仓库目录结构，并定义代码、契约、测试、部署和运行数据的边界。

当前 Java Interview Coach 原型仍需保持可运行，因此目录调整采用渐进迁移。Codex MVP 已建立 `apps/`、`contracts/`、`deploy/`、`skills/` 和 `tools/`，原型资产仍保留原位且不自动迁移。

## 3. 目录职责

### 3.1 `apps/trainer-core`

Training Core 是唯一可以直接访问业务 PostgreSQL 的应用。

首期按模块化单体组织包边界：

```text
trainer-core/
└─ src/main/
   ├─ java/.../mindtrain/
   │  ├─ identity/
   │  ├─ catalog/
   │  ├─ question/
   │  ├─ curation/
   │  ├─ training/
   │  ├─ assessment/
   │  ├─ scheduling/
   │  ├─ integration/
   │  ├─ reporting/
   │  └─ ai/
   └─ resources/
```

模块之间通过应用服务或领域事件协作，不允许跨模块直接访问内部 Repository。首期不把这些模块拆成独立部署服务。

### 3.2 `apps/trainer-mcp`

Trainer MCP 只调用 Core API 和本地 Scheduler Plugin，不直接访问数据库。

它负责：

- 把 API 能力转换为稳定、低泄露风险的 MCP 工具。
- 管理 Core Token、本地插件地址和请求幂等键。
- 根据 Session 的 Provider 把调度调用路由到 Core 或 Anki Bridge。
- 将底层错误转换为 AI 可以安全处理的工具错误。

### 3.3 `apps/web`

Web 应用同时承载学习端和内容管理端，首期保持单一前端工程。它只通过 Core API 访问数据。

### 3.4 `integrations/anki`

Anki 集成独立于 Core 领域实现：

- `addon/` 包含 Anki 内运行的 Python Add-on 和本地 MCP/HTTP Endpoint。
- `bridge-contract/` 定义 Question Projection、Review Rating、Backlog 和同步游标。
- `compatibility/` 保存受支持的 Anki 版本、自动化测试配置和第三方 MCP PoC 记录。

Core 不依赖 Anki Python API；所有 Anki 专有概念都停留在 Adapter 和 Binding 中。

### 3.5 `skills`

- `knowledge-trainer` 是最终领域无关 Skill，只调用 Trainer MCP。
- `java-interview-coach` 在迁移期继续支持现有训练，完成迁移后变为 Java Knowledge Pack 的对话适配或被前者替代。
- Skill 内不保存题库、用户记录或生产 Prompt 数据。

### 3.6 `contracts`

契约是所有客户端和插件共享的稳定边界：

- REST 使用 OpenAPI。
- 异步事件使用 JSON Schema 或 AsyncAPI。
- Knowledge Pack 定义可移植数据结构和版本。
- Scheduler SPI 定义 Provider 的输入输出、错误码、幂等和能力协商。

契约变更必须考虑兼容性，并通过 `tests/contract` 验证。

### 3.7 `database`

- `migrations/` 只保存数据库结构、索引和必要的受控初始化数据。
- 正式题库、候选题、Prompt 和个人学习数据不能作为普通仓库文件长期维护。
- `test-fixtures/` 只保存最小匿名测试数据，不能复制真实用户历史。

### 3.8 `deploy`

每种部署模式独立提供 Compose、环境变量示例和健康检查：

- `core-only`：默认加权调度器。
- `anki-local`：增加 Anki Desktop、Add-on 和 Trainer MCP 配置说明。
- `anki-sync-server`：增加官方同步服务、TLS/VPN 和备份说明。

### 3.9 `tools`

所有一次性或运维工具都必须可重复执行、支持 dry-run，并输出机器可读的数量对账结果。数据迁移工具不能静默覆盖已有 Attempt 或 Question Version。

### 3.10 `prototype`

该目录是迁移期边界，不是长期架构的一部分。它用于冻结当前 Java Skill 原型，直到所有数据完成导入、校验和回滚演练。

迁移完成后：

- 可运行逻辑迁入 `apps/`、`skills/` 和 `tools/`。
- 生产资产迁入 Training Core 实例。
- 必要的匿名样例迁入 `tests/fixtures/`。
- 私人学习记录从 Git 历史和工作目录管理流程中退出。

## 4. 禁止进入仓库的运行数据

以下内容统一写入本地 `var/`、数据库或外部 Secret 管理，并通过 `.gitignore` 排除：

```text
var/
├─ postgres/
├─ object-storage/
├─ exports/
├─ plugin-state/
├─ anki-sync/
└─ logs/
```

禁止提交：

- 真实用户 Attempt、Session、Mistake 和对话记录。
- 生产题库导出、未经处理的个人 Candidate。
- PostgreSQL、Anki Collection 和对象存储数据卷。
- Core Token、模型密钥、Anki 凭据和 Tunnel Token。
- 包含私人学习信息的日志、备份和诊断文件。

## 5. 依赖方向

允许的依赖方向：

```text
Web ────────────────┐
Trainer MCP ────────┼──> Training Core API ──> PostgreSQL
Anki Bridge ────────┘             │
                                  └──> Contracts

Knowledge Trainer Skill ──> Trainer MCP
Anki Bridge ───────────────> Bridge Contract
```

禁止：

- Skill 直接访问 PostgreSQL 或 Anki Collection。
- Trainer MCP 直接读写业务数据库。
- Core 引用 Anki Add-on 的内部 Python 类型。
- Web 调用 Anki 本地接口。
- 测试 Fixture 被当作生产 Knowledge Pack 使用。

## 6. 当前目录到目标目录的映射

| 当前目录 | 目标位置 | 迁移条件 |
| --- | --- | --- |
| `skill/java-interview-coach/` | `skills/java-interview-coach/` | Trainer MCP 能覆盖原型脚本能力后 |
| `schemas/` | `contracts/` 与数据库模型 | 完成 API、事件和导入格式分类后 |
| `assets/` | Training Core 数据库 | 导入、来源校验和数量对账完成后 |
| `learning-data/` | Training Core 用户数据 | Attempt 幂等导入和报告一致性验证后 |
| `prompts/` | Core PromptTemplate 数据 | Prompt 版本和回滚 API 完成后 |
| `evals/` | `tests/ai-evals/` | Eval Runner 能读取新格式后 |
| `scripts/` | `tools/prototype-migration/` 或目标应用 | 按脚本职责拆分后 |
| `tests/` | 新分层的 `tests/` | Core 和契约骨架建立后 |

迁移期间当前目录保持原位，避免同时改变路径和业务行为。

## 7. 分阶段迁移

### 阶段 A：建立骨架

- 创建 `apps/trainer-core`、`contracts`、`database` 和 `deploy/core-only`。
- 建立 Core 健康检查、数据库迁移和 CI。
- 不移动原型目录。

当前状态：Core、Trainer MCP、Flyway Migration、Core-only Compose、OpenAPI、Scheduler Contract 和 Knowledge Trainer Skill 已建立。数据库 Migration 暂随 `trainer-core` 打包，后续如需独立数据库发布物再同步抽取到 `database/migrations/`。

### 阶段 B：迁移权威数据

- 实现 Question、Topic、Source、Prompt、Session 和 Attempt API。
- 建立 `tools/prototype-migration`。
- Dry-run 后导入现有资产并完成数量与校验结果对账。

### 阶段 C：迁移交互与调度

- 建立 Scheduler SPI 和默认 Weighted Provider。
- 实现 Trainer MCP。
- 新建 `skills/knowledge-trainer` 并迁移 Java Skill 工作流。

### 阶段 D：Anki 插件

- 用第三方 MCP Add-on 完成 PoC。
- 实现 `integrations/anki/addon` 和 Bridge Contract。
- 完成真实 Scheduler、FSRS、离线和幂等测试。

### 阶段 E：清理原型

- 停止向仓库文件写入生产数据。
- 将必要原型代码归档到 `prototype/java-interview-coach`。
- 删除已完成迁移且有备份的私人运行数据。
- README 切换为新应用的构建、运行和部署说明。

## 8. 命名约定

- 部署单元使用 kebab-case：`trainer-core`、`trainer-mcp`。
- Java 包使用 `io.mindtrain` 根命名空间。
- REST 资源使用复数名词和 `/api/v1` 前缀。
- 事件使用过去式，例如 `AttemptRecorded`、`ReviewScheduled`。
- Scheduler Provider 使用稳定小写 ID，例如 `weighted`、`anki`。
- 数据库表使用 snake_case。
- 环境变量使用 `MINDTRAIN_` 前缀。

## 9. 当前阶段约束

本次只建立目录规划文档并同步 README，不创建空目录、不移动现有文件、不修改原型运行路径。下一阶段应从 `apps/trainer-core` 和 `contracts` 骨架开始实施。
