# MindTrain 目标需求

## 1. 文档目的

本文定义 MindTrain 从 Java 面试 Codex Skill 原型升级为通用知识训练平台后的目标、范围、核心能力和验收原则。本文描述产品要解决什么问题；具体技术实现见《概要设计》文档。

## 2. 产品定位

MindTrain 是一个 AI 驱动、调度算法可插拔、领域数据可独立管理的知识训练平台。

平台需要同时支持：

- 独立运行的知识训练应用和 API。
- 与 Codex 集成的 AI 教练 Skill。
- 可安装的领域知识包，例如 Java 后端面试知识包。
- 默认训练调度算法。
- 通过插件接入 Anki/FSRS 等外部调度器。
- AI 生成、人工审核、版本管理和持续扩充知识资产。

MindTrain 的核心价值不是重新实现聊天模型或单纯复制 Anki，而是把结构化知识资产、训练过程、AI 教学能力和可替换调度器组合为一个完整系统。

## 3. 目标用户与典型场景

### 3.1 目标用户

- 需要持续准备技术面试的开发者。
- 希望用 AI 对话方式学习专业知识的个人用户。
- 希望建设和维护领域题库的内容维护者。
- 希望把自有知识资产接入训练系统的团队。
- 希望使用 Anki 管理复习计划、同时保留 AI 教练体验的用户。

### 3.2 典型场景

- 用户每天进行固定题量的知识训练，并可在答题过程中随时追问。
- 系统优先安排到期复习，再按容量引入新知识。
- AI 根据知识缺口生成临时题，经 Core 验证后入库展示；用户作答后进入普通调度，拒绝则物理删除。
- 用户选择默认调度器，或连接本地 Anki 使用其 FSRS 调度。
- 内容维护者导入、修订、升级和删除知识资产。
- 管理者查看覆盖率、正确率、遗忘风险、积压和训练负担。

## 4. 核心产品目标

### 4.1 通用知识训练

- 领域模型不得绑定 Java；Java 后端仅作为首个知识包。
- 支持领域、知识点、关系、题目、来源、解释和版本等结构化资产。
- 首期继续支持四选项单选题和多选题，后续允许扩展简答、判断、代码分析等题型。
- 每个训练 Session 必须绑定一个有效领域；单领域实例可自动选择，多领域实例必须由用户明确选择，禁止写死或隐式选取某个领域。
- 复习、新题、AI 临时题和调度积压都必须按 Session 领域隔离。
- 单选和多选继续采用确定性判分；AI 负责教学解释，不能覆盖确定性结果。

### 4.2 AI 教练体验

- 用户应在同一对话中完成出题、答题、解释、追问、反例和深入训练。
- 用户的临时追问不得被误认为正式答案或自动消耗题目。
- 展示题目时不得泄露正确选项数量或答案组合。
- 答题提示统一使用中性文本，例如“请回复选项字母，可用逗号分隔”。
- AI 生成内容必须记录模型、Prompt 版本、来源和生成时间。

### 4.3 统一权威数据服务

- 题库、AI 临时题、知识分类、来源、Prompt、用户、会话、答题、掌握度和调度绑定全部由应用服务持久化。
- 生产运行数据不得继续以仓库内 JSON/JSONL 文件作为权威存储。
- 代码仓库只保留协议、迁移、应用代码、部署配置和少量测试 Fixture。
- 支持完整知识包导入导出，使其他实例可以独立安装和管理数据。
- 支持单用户部署，并为后续多用户和多租户保留清晰的数据归属字段。

### 4.4 可插拔调度

- 提供默认的加权调度算法，综合到期、薄弱程度、错误频率、覆盖率和重要性。
- 调度算法必须通过统一接口接入，不能写死在 Skill 或控制器中。
- 支持 Anki 调度插件，由 Anki/FSRS 决定已经学习过的卡片何时复习。
- Training Core 决定哪些新题可以进入外部调度器，避免一次性导入全部题库。
- 更换调度器时，题库、Attempt、Session 和 Review Event 不得丢失。

### 4.5 有限训练容量

- 用户可设置每日或单次训练预算，默认一次 10 题。
- 系统必须区分复习预算、新题预算和重新学习预算。
- 到期积压过高时可自动暂停新题引入。
- 默认建议以复习为主、少量引入新题；具体比例可配置并根据积压动态调整。
- 系统必须展示 Due Backlog、最老逾期时间和预计训练负担。

### 4.6 Anki 集成

- Anki 定位为可选调度插件和本地数据投影，不是 MindTrain 的权威数据库。
- 提供 MindTrain 专用 Anki Add-on/Bridge，负责卡片投影、真实 Scheduler 取题、评级和事件同步。
- 提供适配 Codex 的 MCP 工具，并说明其本地运行、鉴权和网络依赖。
- Codex Plugin 与 Trainer MCP 必须进行双向版本和契约兼容性校验；不兼容时阻止远程训练并给出明确升级指引，组件版本不同但契约兼容时只提醒同步升级。
- 同一道题在 Anki 中必须保持稳定 Card 映射，更新内容时不得无故重建 Card。
- Anki 评级必须通过其 Scheduler 提交，不能由 MindTrain 手工修改 Due Date 替代。
- AnkiWeb 或自托管 Sync Server 仅用于 Anki 多端同步，不承担 MindTrain API 职责。

### 4.7 内容治理

- AI 生成题在展示前先以临时题入库，并由 Core 完成 Schema、来源、题型、难度和知识点校验。
- 用户未作答前可明确拒绝 AI 临时题；Core 必须物理删除待答 Assignment、QuestionVersion 和 Question，并恢复本轮新题额度。
- AI 临时题一旦作答，立即转为可跨 Session 调度的普通题，不需要额外人工发布。
- 已作答题不允许通过“拒绝”删除学习历史；内容或评分标准变更必须递增版本。
- 未作答临时题在 Session 结束或超时后自动物理清理。
- 每道持久题必须保留来源 URL、访问时间和适用版本。
- 用户可以命名一个只在 Codex 主机保存的本地资料目录，由 Codex 按用户选择的内容整理训练领域、知识点层级、关系和来源绑定。
- 用户也可以通过 AI 对话说明学习目标、范围、受众和难度，由 Codex 规划新的训练领域或扩展已有领域；无参考资料时预览必须提示来源未覆盖。
- 本地原文件、抽取文本、绝对路径与检索索引不得上传服务端；Core 只保存来源元数据、哈希、用户确认的训练结构和题目。
- 训练领域和知识点保存前必须在 Codex 对话中完整预览，并由用户一次明确确认；Core 不替用户决定学习内容。
- 一个实例支持多个领域，一个领域支持多个根知识点；Codex 和 Web 均可通过认证接口进行树形查询、搜索和详情查看。

## 5. 系统角色

| 角色 | 主要权限 |
| --- | --- |
| 学习者 | 创建训练、答题、追问、选择调度器、查看报告 |
| 内容维护者 | 修订已作答题版本、管理来源和知识包 |
| 实例管理员 | 管理知识包、模型配置、插件、用户和部署设置 |
| 调度插件 | 获取必要题目投影、返回调度结果和 Review Event |
| AI 教练 | 生成临时题、解释答案、组织追问，不直接篡改权威数据 |

## 6. 关键数据归属

### 6.1 MindTrain 权威数据

- Knowledge Domain、Topic、Topic Relation。
- Question、Question Version、Option、Explanation。
- AI 临时题、Source、Prompt Template、Eval Case。
- User、Training Session、Assignment、Attempt、Mistake。
- Review Event、Mastery Aggregate、Scheduler Profile、Plugin Binding。
- AI 生成记录、审核记录、同步状态和幂等记录。

### 6.2 Anki 数据投影

- MindTrain Question ID 和版本。
- 题目正反面渲染所需字段。
- Anki Note/Card ID 映射。
- 本地 FSRS 状态、Due Date 和 Revlog。
- 与 MindTrain 同步所需的版本及事件游标。

### 6.3 Codex Skill 数据

Skill 不拥有业务数据，只包含：

- 何时调用哪些工具的工作流。
- 题目展示和解释顺序。
- 失败恢复、追问和会话协议。
- Core MCP 与 Anki MCP 的使用约束。

## 7. 调度与新题引入原则

默认数据库配置为每轮 10 题和 2 道计划内新题，复习题数量由 `questionCount - newBudget` 得到 8 题：

- Learning/Relearning 和严重逾期题优先。
- 到期题不足时，使用已有未学习题或 AI 生成题补足总预算；超出计划内新题数时标记为 shortage fill。
- Due Backlog 超过 20 或最老逾期超过 3 天时，默认暂停新题。
- 新题引入量必须根据近期积压和预计未来负担动态降低。
- 用户可以手动覆盖当天策略，但系统应显示负担风险。

上述数字是首版默认值，必须配置化，不能成为领域协议的固定常量。

## 8. 部署目标

### 8.1 Core-only 模式

- Training Core。
- PostgreSQL。
- 可选对象存储。
- Trainer MCP。
- Codex Skill 或 Web 客户端。
- 使用默认加权调度器。

### 8.2 Anki 集成模式

在 Core-only 基础上增加：

- Anki Desktop。
- MindTrain Anki Bridge Add-on。
- 本地 Anki MCP/HTTP 接口。
- 可选 AnkiWeb 同步。

### 8.3 私有化 Anki 同步模式

进一步增加：

- 官方 Anki Sync Server。
- HTTPS 反向代理或 VPN。
- 独立持久化卷和备份。

Anki Sync Server 与 Training Core 必须独立部署和独立备份。

## 9. 非功能要求

### 9.1 数据一致性

- 所有写操作提供幂等键。
- Attempt 历史追加保存，不静默改写。
- 插件同步失败进入可重试状态，并可人工查看。
- Anki 评级成功但 Core 写入失败，或反向失败时，都必须可恢复。

### 9.2 安全与隐私

- API Token、Anki 凭据和模型密钥不得进入仓库。
- 本地 MCP 默认只监听 loopback；远程暴露必须启用鉴权和 TLS。
- 用户可以导出和删除个人学习数据。
- AI 调用必须记录发送的数据类型和供应商。

### 9.3 可移植性

- Core API 和知识包格式不得依赖 Codex。
- 调度协议不得依赖 Anki 专有字段。
- 知识包能够在不同 MindTrain 实例间导入导出。
- Java Knowledge Pack 可以独立升级，不要求升级平台核心。

### 9.4 可测试性

- 判分、默认调度、容量控制和幂等逻辑必须有确定性测试。
- Scheduler Provider 必须通过统一契约测试。
- Anki Bridge 需要真实 Anki 版本兼容测试。
- Prompt 和生成质量需要固定 Eval Case 回归。

## 10. 首阶段非目标

- 不在首版建设微服务集群；Training Core 采用模块化单体。
- 不在首版实现复杂组织权限和商业题库市场。
- 不要求 Anki 无客户端、纯服务器运行。
- 不将未作答 AI 临时题暴露给其他 Session。
- 不让 AI 自行删除已作答题或学习历史；只有用户明确拒绝当前未答 AI 题时才允许物理删除。
- 不在首版实现所有题型和所有模型供应商。

## 11. 平台基线

- 仓库、README 和设计文档统一使用 MindTrain 名称与平台定位。
- 目标需求和概要设计明确 Training Core、Skill、MCP、Scheduler SPI 和 Anki Bridge 边界。
- 生产数据只保存在服务实例中，仓库不保留题库、个人学习历史或迁移期 JSON/JSONL 数据。
- 默认调度和 Anki 调度能够在同一抽象下描述。
- 文档给出每天 10 题时的容量与积压策略。
- Plugin 内的 MindTrain Skill 是唯一发布版本，通过 Trainer MCP 使用权威应用数据。
