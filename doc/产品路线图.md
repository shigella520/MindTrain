# MindTrain 产品路线图

## 1. 文档目的

本文描述 MindTrain 在 1.0 之后的发展方向、优先级和阶段边界。它回答“接下来优先解决什么问题”；产品能力与验收原则见[目标需求](目标需求.md)，技术边界见[概要设计](概要设计.md)。

路线图按用户成果和依赖关系排序，不承诺固定发布日期。相邻阶段可以并行探索，但依赖后续能力的功能不能跳过前置的数据边界和验收条件直接落地。

## 2. 长期定位

MindTrain 是面向个人和小型团队的私有 AI 知识训练平台。用户决定学习什么并拥有训练领域、题目、来源和学习记录；AI 负责辅助整理、出题和讲解，Training Core 负责确定性判分、持久化和调度。

MindTrain 核心不绑定 Java 或其他具体知识领域，也不预装官方题库。训练内容由用户通过 AI 对话、本地资料或未来的导入功能创建，并在确认后成为其私有实例中的数据。

## 3. 发展原则

1. **先完成训练闭环，再扩大平台范围**：近期优先解决题目是否可管理、复习是否清晰、历史是否可追踪。
2. **用户拥有内容和学习数据**：训练领域与学习记录必须可备份、恢复和迁移，且两类数据分别管理。
3. **AI 辅助，Core 决策**：AI 不直接判分或绕过确认修改权威数据。
4. **单用户私有部署优先**：多用户能力建立在数据归属、权限和审计成熟之后。
5. **外部集成后置且可替换**：先稳定 Core 调度与事件模型，再接入 Anki/FSRS；外部系统不成为题库权威来源。
6. **保持模块化单体**：在模块边界和部署需求没有证明微服务必要性以前，不拆分服务。

## 4. 当前基线：1.0（已完成）

- Training Core、Trainer MCP、MindTrain Web 和 Codex Plugin 可独立部署。
- 支持通过 AI 对话或本地资料创建多个训练领域及知识点树。
- 一个 Session 严格绑定一个领域，复习题、新题和 AI 临时题按领域隔离。
- 支持选择题精确判分、追问、提示、质疑、AI 补题和作答记录。
- 提供加权调度、积压控制、掌握度、薄弱与擅长内容以及动态 Dashboard。
- 支持 Plugin 与服务端契约版本检查。

## 5. 分阶段路线

### 阶段 A：可管理的个人训练闭环（近期）

目标：用户不仅能训练，还能看清并管理训练过程中产生的内容和记录。

- 建立题库管理入口：搜索、筛选、题目详情、来源、版本和状态查询。
- 提供待复习、错题、薄弱知识点和擅长知识点的可下钻列表。
- 提供近期 Session、单次训练详情和历史趋势。
- 完善题目修订、来源治理、失效标记和审计信息。
- 统一 Web 与 Codex 对核心训练、目录查询和报告能力的表达。

完成标志：用户能从 Dashboard 指标进入对应明细，定位具体题目、知识点和训练记录，并完成受控治理。

### 阶段 B：数据安全与可移植性（近期至中期）

目标：私有实例可以放心升级、恢复和迁移，不因部署操作丢失积累。

- 提供数据库备份、恢复验证、升级前检查和失败回滚指引。
- 分别导出训练内容与个人学习记录，避免把可分享内容和私人数据混为一体。
- 支持训练领域、知识点、题目、来源及版本的预览式导入导出。
- 导入时报告冲突、引用缺失和版本差异，用户确认后再写入。
- 提供实例健康、版本兼容和数据迁移状态诊断。

完成标志：用户可以在新实例恢复完整学习状态，也可以只迁移某个训练领域而不携带私人学习记录。

### 阶段 C：内容质量与训练形式扩展（中期）

目标：提高内容可信度与覆盖能力，并逐步超越选择题训练。

- 提供来源覆盖率、资料新鲜度、待复核内容和无来源题目治理。
- 支持本地资料增量同步、变更检测和知识点来源追踪。
- 支持领域归档、复制、合并预览和知识树结构调整。
- 在保持可验证评分的前提下，逐步增加判断题、简答题或代码分析题。
- 建立生成题质量评测、Prompt 回归和人工复核工作流。

完成标志：新增题型具备明确评分契约，内容变更可追溯，资料更新不会静默破坏既有题目和学习历史。

### 阶段 D：调度器生态（中长期）

目标：在不改变权威数据归属的前提下，为用户提供更成熟或可替换的复习算法。

- 持续校准 Core 加权调度的负担预测、新题准入和重新学习策略。
- 稳定 Scheduler Provider 契约与统一 Review Event。
- 验证 FSRS Provider，并提供调度迁移与对账工具。
- 实现可选的 MindTrain Anki Bridge、稳定卡片映射和失败恢复。
- 建立支持版本矩阵、能力协商和真实客户端兼容测试。

完成标志：更换调度器不会丢失题目、Session、Attempt 或复习事件，外部投影可从 Core 重建。

### 阶段 E：多人协作与共享治理（长期）

目标：在保持个人学习数据隔离的同时，让团队共同维护训练内容。

- 支持多用户身份、权限、审计和配额。
- 区分共享训练领域与个人学习状态；共享内容不共享私人作答记录。
- 支持领域维护者、审核者和学习者工作流。
- 提供团队级内容质量、覆盖情况和匿名聚合洞察。
- 在需求明确后评估共享内容发布机制，不预设公共题库市场。

完成标志：多个用户能够共同治理内容，同时任何用户都不能越权读取或修改他人的学习记录。

## 6. 明确不做或暂不优先

- 不维护平台内置的官方通用题库；示例 Fixture 只用于测试和演示。
- 不把用户创建的训练领域称为平台自带 `Knowledge Pack`。
- 不在近期实现跨领域混合 Session。
- 不把 Anki、Codex Skill 或本地资料索引变成权威数据库。
- 不为尚未实现的功能提前创建空模块和占位契约。
- 不在单用户数据边界稳定前引入复杂组织、多租户或内容市场。

## 7. 路线图维护规则

- README 只展示阶段摘要，并链接本文。
- `目标需求.md` 记录稳定产品原则，不堆叠版本待办。
- `概要设计.md` 的演进里程碑必须与本文阶段顺序一致。
- 每次正式 Release 根据已交付结果更新“当前基线”和阶段状态。
- 新需求优先归入现有阶段；只有形成独立用户成果和明确依赖时才新增阶段。
