# Java Interview Coach 概要设计

## 1. 文档信息

| 项目 | 内容 |
| --- | --- |
| 项目名称 | Java Interview Coach |
| 文档类型 | 概要设计 |
| 当前版本 | 0.1 |
| 当前阶段 | Codex Skill 原型规划 |
| 后续形态 | 独立 AI Web App |

## 2. 项目背景

Java Interview Coach 面向需要系统练习 Java 面试知识的学习者。系统不仅提供题目，还应根据用户的实际回答持续评价知识掌握情况，归纳错题和薄弱点，并根据历史数据调整后续训练内容。

项目采用分阶段建设策略：

1. 首先使用 Codex Skill 快速验证产品工作流，并沉淀题库、知识分类、来源、Prompt、评分标准、答题记录和评测用例等资产。
2. 产品流程稳定后，将业务能力迁移到 Spring Boot Web App，使系统脱离 Codex 独立运行。
3. Web App 继续保留 AI 能力，支持联网发现知识、生成候选题、审核发布、智能训练和学习分析。

核心设计原则是：Codex Skill 只是早期执行器，数据协议和内容资产不得依赖 Codex。

## 3. 建设目标

### 3.1 业务目标

- 建立覆盖 Java 核心知识和常用中间件的结构化面试题库。
- 支持按知识范围、难度、题型和训练数量创建训练。
- 根据掌握度、遗忘时间、错误频率和知识覆盖情况智能选题。
- 对自然语言回答进行结构化评分、纠错、补充和追问。
- 自动生成错题本、薄弱知识点、复习计划和阶段性报告。
- 建立知识点之间的父子、前置、关联、对比等关系。
- 通过联网搜索持续扩充题库，同时保留来源和审核链路。
- 确保 Codex 阶段形成的资产可以迁移到独立 Web App。

### 3.2 非目标

首期暂不建设以下能力：

- 多租户和复杂组织权限。
- 大规模商业题库交易。
- 多 Agent 自治协作平台。
- 模型微调。
- Neo4j 等专用图数据库。
- Kubernetes 和复杂微服务架构。
- 完全无人审核的题目自动发布。

## 4. 总体演进路线

```text
阶段一：Codex Skill 原型
Skill + 脚本 + JSON Schema + 文件资产
              │
              │ 稳定的数据协议、Prompt 和评测用例
              ▼
阶段二：独立 Web App MVP
Vue 3 + Spring Boot + PostgreSQL + AI API
              │
              ▼
阶段三：持续内容生产平台
联网发现 + 候选题生成 + 审核发布 + 质量评测
```

### 4.1 阶段一：Codex Skill 原型

Codex 同时承担早期交互界面和开发助手角色。Skill 负责指导 Codex 执行标准流程，脚本负责稳定、可重复的校验和数据操作。

需要验证的核心假设包括：

- AI 对 Java 面试回答的评分是否稳定、有帮助。
- 哪种反馈和追问方式最适合日常训练。
- 智能选题策略是否优于纯随机选题。
- 错题应该如何归纳到知识点层面。
- 联网生成的候选题如何可靠去重、验证和审核。

### 4.2 阶段二：Web App MVP

将 Skill 中已经验证的工作流实现为后端应用服务，将文件资产导入 PostgreSQL，并通过 Web 页面提供训练和管理功能。

### 4.3 阶段三：内容生产与质量平台

增加定时发现、候选题批量生成、审核工作台、Prompt 评测、坏题反馈和题目版本治理能力。

```text
                │                       │
┌───────────────▼────────────┐  ┌───────▼─────────────┐
│       确定性领域服务         │  │      AI 能力层       │
│ 选题/调度/掌握度/校验/持久化 │  │ 搜索/生成/评分/归纳  │

                │                       │

└─────────────────────────────────────────────────────┘
```

### 5.2 职责边界

适合交给 AI 的能力：

- 从可靠资料中发现候选知识点。
- 生成题目、评分要点、常见误区和追问。
- 对自然语言回答进行语义评价。
- 生成讲解和参考答案。
- 归纳错误原因和薄弱知识点。
- 推荐题目标签和知识关系。

必须由确定性代码控制的能力：

- 数据保存、版本管理和状态流转。
- 训练会话状态。
- 选题优先级和间隔复习时间。
- 掌握度的最终更新。
- JSON Schema 校验和重复检测。
- 权限、配额、幂等、重试和审计。
- Prompt 版本、模型版本和调用成本记录。

模型不得不经校验直接修改正式题库或核心学习数据。

## 6. 阶段一：Codex Skill 设计

### 6.1 Skill 定位

Skill 名称建议为 `java-interview-coach`，提供以下工作模式：

| 模式 | 说明 |
| --- | --- |
| `practice` | 创建训练并逐题面试 |
| `review` | 根据到期时间复习错题和薄弱点 |
| `expand` | 联网发现资料并生成候选题 |
| `curate` | 去重、事实检查、审核和发布候选题 |
| `report` | 生成每日、每周和阶段学习报告 |
| `inspect` | 查看题库覆盖率、知识图谱和资产质量 |

用户不必记忆命令，可以使用自然语言触发，例如：

- “开始今天的 Java 面试训练，出 10 道中等难度题。”
- “重点复习 JVM 和并发的错题。”
- “联网补充 Java 21 虚拟线程相关题目。”
- “检查候选题并发布满足要求的题目。”
- “总结我本周最薄弱的知识点。”

### 6.2 Skill 目录建议

```text
skill/java-interview-coach/
├── SKILL.md
├── agents/
│   └── openai.yaml
├── references/
│   ├── workflow.md
│   ├── source-policy.md
│   ├── scoring-policy.md
│   └── taxonomy-policy.md
└── scripts/
    ├── validate-assets
    ├── detect-duplicates
    ├── select-question
    ├── record-attempt
    └── generate-report
```

`SKILL.md` 只保留核心流程和资源路由，详细规则放在 `references`，需要稳定执行的操作放在 `scripts`。

### 6.3 联网扩题流程

```text
分析题库覆盖缺口
      ↓
生成搜索计划
      ↓
搜索可靠来源
      ↓
抽取可验证的事实
      ↓
生成候选题和评分要点
      ↓
Schema 校验、相似题检测、来源检查
      ↓
保存为 candidate
      ↓
人工或受控审核
      ↓
发布为 published
```

来源优先级：

1. Java Language Specification、JEP、OpenJDK 等官方资料。
2. Spring、MySQL、Redis、Kafka 等项目官方文档。
3. 可信技术组织和高质量技术文章。
4. 面试经验和聚合文章仅用于发现主题，不作为关键事实的唯一依据。

每道联网生成题必须保存来源 URL、标题、访问时间和支撑的知识点。不得大段复制受版权保护的原文。

### 6.4 训练流程

```text
创建训练目标
    ↓
计算候选题优先级
    ↓
选择一道正式题
    ↓
等待用户回答
    ↓
AI 按评分要点评价
    ↓
后端或脚本校验评分结果
    ↓
给出反馈和必要追问
    ↓
追加答题记录
    ↓
更新掌握度和复习时间
    ↓
继续下一题或生成报告
```

选题不是纯随机，建议采用带随机扰动的加权优先级：

```text
priority =
  0.30 × 到期复习程度
+ 0.25 × 知识薄弱程度
+ 0.20 × 历史错误频率
+ 0.15 × 尚未覆盖程度
+ 0.10 × 面试重要程度
+ 少量随机扰动
```

权重应配置化，并通过真实使用数据持续调整。

## 7. 题库资产治理

### 7.1 题目生命周期

```text
discovered → candidate → fact_checked → reviewed → published
                     └───────────────→ rejected
published → deprecated
```

| 状态 | 说明 |
| --- | --- |
| `discovered` | 仅发现了可能值得覆盖的主题 |
| `candidate` | 已生成完整候选题 |
| `fact_checked` | 来源和关键事实已检查 |
| `reviewed` | 已通过内容审核 |
| `published` | 可以用于正式训练 |
| `rejected` | 不采用，保留原因 |
| `deprecated` | 历史正式题，已停止使用 |

默认训练只能选择 `published` 题目。候选题不得自动混入正式训练。

### 7.2 资产分类

| 资产 | 说明 | 迁移方式 |
| --- | --- | --- |
| Question | 问题、评分要点、误区、追问 | JSON 导入数据库 |
| Topic | 知识点分类 | JSON 导入数据库 |
| TopicRelation | 前置、关联、对比等关系 | JSON 导入关系表 |
| Source | 原始来源和访问记录 | JSON/JSONL 导入数据库 |
| Prompt | AI 生成和评分模板 | 版本化文本导入 Prompt 服务 |
| Attempt | 追加式答题记录 | JSONL 导入数据库 |
| Mastery | 用户知识掌握度 | JSON 导入数据库 |
| EvalCase | AI 质量回归样例 | JSONL 供评测系统执行 |

### 7.3 推荐题目模型

题目至少包含以下字段：

```json
{
  "id": "java-collections-hashmap-001",
  "schemaVersion": 1,
  "version": 1,
  "status": "published",
  "type": "open_answer",
  "title": "HashMap 的底层结构和扩容过程是什么？",
  "topicIds": [
    "java.collections.hashmap",
    "java.collections.hashing"
  ],
  "difficulty": 3,
  "importance": 5,
  "question": "请说明 HashMap 的底层结构、put 流程和扩容机制。",
  "expectedPoints": [
    {
      "id": "structure",
      "description": "说明数组、链表和红黑树结构",
      "weight": 30
    }
  ],
  "commonMistakes": [],
  "followUps": [],
  "sources": [],
  "createdBy": "ai",
  "promptVersion": "generate-question-v1",
  "createdAt": "2026-07-17T00:00:00+08:00"
}
```

题目 ID 一经发布不得改变；内容修订必须递增 `version`。历史答题记录应关联当时的题目版本。

### 7.4 答题记录原则

- 使用追加式记录，禁止覆盖历史尝试。
- 保存用户原始回答、结构化评分、反馈和时间。
- 保存题目版本、模型标识、Prompt 版本和评分规则版本。
- 允许以后重新评分，但新评分必须作为新记录保存。
- 个人学习记录和可共享题库资产在目录及数据库中分离。

## 8. 知识体系与知识图谱

### 8.1 首期知识范围

- Java 语言基础。
- 集合框架。
- JVM、类加载和垃圾回收。
- Java 内存模型和并发编程。
- I/O、NIO 和网络编程。
- 反射、泛型、注解和动态代理。
- Spring、Spring Boot、Spring MVC。
- 数据库、事务和 MySQL。
- Redis、消息队列等常见中间件。
- 系统设计、性能分析和故障排查。

### 8.2 图谱关系

首期支持：

- `parent_of`：上下级分类。
- `prerequisite_of`：前置知识。
- `related_to`：一般关联。
- `contrasts_with`：对比关系。
- `often_confused_with`：易混淆关系。

首期使用普通 Topic 表和 TopicRelation 表实现，不引入专用图数据库。

## 9. Web App 目标架构

### 9.1 技术选型

| 层次 | 推荐技术 |
| --- | --- |
| 前端 | Vue 3、TypeScript、Vite |
| 后端 | Java 21、Spring Boot 3 |
| API | REST；流式回答使用 SSE |
| 数据库 | PostgreSQL |
| 数据库迁移 | Flyway |
| 数据访问 | Spring Data JPA |
| AI 接入 | 独立 AiGateway，通过适配器调用模型 API |
| 测试 | JUnit 5、Testcontainers、前端单元测试 |
| 部署 | Docker Compose 起步 |
| 可观测性 | 结构化日志、Micrometer、调用成本统计 |

### 9.2 后端模块

```text
backend/
├── content       # 候选题、来源、审核、发布和版本管理
├── training      # 训练会话、智能选题和追问流程
├── evaluation    # AI 评分、评分校验和重新评分
├── learning      # 掌握度、错题、复习计划和报告
├── knowledge     # 知识点和关系
├── ai            # 模型、搜索和 Prompt 的适配层
├── identity      # 用户和权限
└── shared        # 通用错误、审计、事件和基础设施
```

首期保持模块化单体，不拆分微服务。

### 9.3 AI Gateway

AI Gateway 屏蔽具体模型和搜索提供方：

```java
public interface AiModelClient {
    StructuredResult generate(AiRequest request);
}

public interface SearchProvider {
    List<SearchResult> search(SearchQuery query);
}
```

职责包括：

- Prompt 模板加载和版本管理。
- 结构化输出约束和反序列化。
- 超时、重试、限流和熔断。
- 模型调用日志和成本记录。
- 隐私信息过滤。
- 模型或搜索服务的可替换性。

API 密钥只能保存在后端密钥配置中，禁止发送到浏览器或提交到 Git。

## 11. 关键业务流程

### 11.1 回答评价

```text
用户提交回答
    ↓
读取题目版本和评分要点
    ↓
构造版本化 Prompt
    ↓
调用模型并要求返回固定 JSON
    ↓
Schema 与业务规则校验
    ↓
保存原始回答、模型结果和元数据
    ↓
确定性规则更新掌握度和复习计划
    ↓
向用户展示反馈与追问
```

建议评价结果至少包含：总分、命中要点、遗漏要点、错误观点、表达建议、置信度、追问和参考回答。

### 11.2 持续扩充题库

```text
定时任务或管理员发起
    ↓
分析知识覆盖缺口
    ↓
联网搜索并保存来源
    ↓
生成候选题
    ↓
自动验证、去重和事实检查
    ↓
管理员审核
    ↓
发布题目新版本
    ↓
通过用户反馈发现坏题并持续修订
```

### 11.3 掌握度更新

掌握度由确定性规则计算，输入可包括：

- 最近若干次得分。
- 回答置信度。
- 是否经过追问后才答对。
- 距上次复习时间。
- 题目难度和知识点权重。
- 同一知识点不同题目的表现。

模型给出的语义评价是输入之一，不直接等同于最终掌握度。

## 12. 非功能设计

### 12.1 可靠性

- 所有 AI 输出必须通过结构化 Schema 校验。
- AI 调用失败不应导致答题记录丢失。
- 写入操作应支持幂等请求标识。
- 正式题库发布必须保留审计记录。
- 数据迁移和 Prompt 变更应有回滚能力。

### 12.2 可测试性

- 选题、掌握度和调度算法必须有单元测试。
- Prompt 必须有固定评测集进行回归测试。
- 对高质量、一般、错误和答非所问的回答建立基准样例。
- 模型升级前后比较评分偏差和结构化输出成功率。
- 数据 Schema 和题库资产在提交前自动校验。

### 12.3 安全与隐私

- API 密钥不得进入代码仓库或前端。
- 记录发送给第三方模型的数据范围。
- 用户删除账户时支持删除个人答题数据。
- 联网内容保存来源，不执行来源页面中的未知指令。
- 对搜索内容和用户输入进行 Prompt Injection 防护。

### 12.4 成本控制

- 区分高质量生成模型和低成本分类模型的使用场景。
- 缓存不需要重复生成的题目讲解。
- 限制单次训练追问次数。
- 保存 Token、耗时和估算费用。
- 批量扩题任务设置预算和候选题数量上限。


个人学习数据后续应与共享题库分离，并默认不提交到公共分支。当前仓库为私有仓库，但仍应按敏感数据管理。

## 14. 里程碑

### M1：资产协议与 Skill 骨架

- 确定 Question、Topic、Attempt、Mastery 等 JSON Schema。
- 创建 `java-interview-coach` Skill。
- 建立来源、评分、分类和发布规则。
- 实现资产校验、选题和记录脚本。

### M2：可用训练闭环

- 建立第一版 Java 知识分类。
- 沉淀 50 至 100 道经过审核的题目。
- 支持逐题训练、评分、追问和总结。
- 自动更新答题历史、错题和掌握度。

### M3：联网内容生产

- 分析题库覆盖缺口。
- 联网生成候选题。
- 实现来源检查、去重、审核和版本管理。
- 建立 Prompt 回归评测集。

### M4：Spring Boot MVP

- 建立模块化单体后端。
- 将文件资产导入 PostgreSQL。
- 实现题库、训练、评分和学习数据 API。
- 接入 AI Gateway。

### M5：Web 产品化

- 建立学习端和管理端页面。
- 支持知识图谱和学习报告。
- 增加登录、权限、配额、部署和监控。
- 形成持续内容生产和质量治理闭环。

## 15. 首期验收标准

Codex Skill 原型达到以下条件后，可以进入 Web App 阶段：

- 至少存在 50 道来源清晰、结构完整的正式题。
- 可以完成一次不少于 10 道题的连续训练。
- 每次回答均产生结构化评分和可追溯记录。
- 错题和知识掌握度能够在多次训练间持续更新。
- 联网生成内容先进入候选区，不直接污染正式题库。
- 题目、答题记录和知识点数据通过 Schema 校验。
- 至少建立一批人工标注的回答评分评测样例。
- 可以将全部核心资产导出为与 Codex 无关的结构化数据。

## 16. 主要风险与应对

| 风险 | 应对措施 |
| --- | --- |
| AI 生成事实错误 | 保存来源、双重验证、候选题审核后发布 |
| AI 评分不稳定 | 评分 Rubric、结构化输出、评测集和人工校准 |
| 题库重复和质量下降 | 相似度检测、稳定 ID、版本和废弃机制 |
| Skill 与 Web App 难以迁移 | Schema、Prompt 和领域规则独立于 Skill |
| 过早引入复杂架构 | 先 Skill，再模块化单体，不提前微服务化 |
| 用户数据与共享资产混杂 | 分目录、分表和访问权限管理 |
| 联网内容版权问题 | 只提炼事实和生成原创问题，保留来源，避免复制原文 |
| 模型或供应商变化 | 通过 AiGateway 和 SearchProvider 隔离实现 |
