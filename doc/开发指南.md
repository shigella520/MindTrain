# MindTrain 开发指南

本文面向希望修改 Training Core、Trainer MCP、Web、Skill 或部署配置的开发者。普通用户请从仓库 [README](../README.md) 的快速开始进入。

## 1. 技术栈

| 模块 | 技术 |
| --- | --- |
| Training Core | Java 21、Spring Boot、Maven、PostgreSQL、Flyway |
| Trainer MCP | Java 21、Spring Boot、Streamable HTTP MCP |
| MindTrain Web | Vue 3、TypeScript、Vite、Vue Router、Pinia |
| Codex Plugin | Marketplace manifest、MindTrain Skill、Python 本地 MCP bridge |

## 2. 本地依赖

- Java 21
- Maven 3.9+
- Python 3.12+
- Node.js 22+
- pnpm 10+
- Docker 与 Docker Compose

## 3. 仓库模块

```text
MindTrain/
├── apps/
│   ├── trainer-core/       # 权威业务数据、判分、调度和报告
│   ├── trainer-mcp/        # MCP Gateway，只访问 Core API
│   └── web/                # Dashboard、Web 训练和管理页
├── plugins/mindtrain/      # Codex Plugin、bridge 和 bundled Skill
├── contracts/              # OpenAPI 与 Scheduler SPI
├── deploy/core-only/       # Docker Compose 私有部署
├── doc/                    # 产品、设计和使用文档
└── tests/                  # Plugin 检查；Core/Web 测试位于各自模块
```

更完整的职责与依赖规则见[仓库目录规划](仓库目录规划.md)。

## 4. 后端验证

从仓库根目录运行：

```bash
mvn -B verify
```

数据库结构变更必须新增 Flyway migration，不要修改已发布 migration。REST 契约变更应同步更新：

```text
contracts/openapi/trainer-core.yaml
```

## 5. Web 开发

```bash
cd apps/web
pnpm install
pnpm dev
```

测试与生产构建：

```bash
pnpm test
pnpm build
```

开发服务器将 `/api` 代理到本地 Training Core。生产镜像使用 Nginx 提供静态资源，并将同源 `/api/*` 转发给 `trainer-core:8080`。

Web 的视觉和交互约定见 [Web 设计](Web设计.md)。

## 6. Plugin Skill 与仓库检查

```bash
make bootstrap
make check
```

必须保持以下约束：

- 单选和多选按选项集合完全相等判分，分数只有 `0` 或 `100`。
- 提交答案前不能泄露正确选项数量、组合或解释。
- 中性答题提示使用“请回复选项字母，可用逗号分隔”。
- JSON、YAML 和 OpenAPI 契约在提交前必须校验。
- 历史 Attempt 追加保存，不能静默覆盖。
- 私人学习数据、Token 和候选题不能进入普通提交。
- Plugin 内的 `plugins/mindtrain/skills/mindtrain` 是 MindTrain Skill 的唯一源码，不维护第二份仓库级副本。

## 7. Codex Plugin 开发

Plugin 位于：

```text
plugins/mindtrain/
```

Marketplace 清单位于：

```text
.agents/plugins/marketplace.json
```

Plugin 或 Skill 更新后应完成 manifest、Skill 和 MCP 工具契约校验，并通过 cachebuster 触发 Codex 重新安装。安装、更新以及 `main` / 短期功能分支切换见 [Codex Plugin 部署](CodexPlugin部署.md)。

## 8. Docker 镜像与 CI

GitHub Actions 构建：

- `ghcr.io/shigella520/mindtrain-core`
- `ghcr.io/shigella520/mindtrain-mcp`
- `ghcr.io/shigella520/mindtrain-web`

发布规则：

| 触发方式 | 行为 |
| --- | --- |
| Pull Request → `main` | 运行 Java、Web、Plugin 验证并试构建镜像，不推送 |
| Push → `main` | 运行完整验证并试构建镜像，不推送 |
| Push Tag → `vX.Y.Z` | 验证并发布 `X.Y.Z`、`latest` 与提交 SHA 标签 |
| 从任意分支手动运行 `Publish Dev Images` | 验证并构建所选分支，发布 `dev` 与提交 SHA 标签 |

正式软件版本、Git Tag、Plugin 版本和 Release 镜像标签的统一维护方式见[版本与发布规范](版本与发布规范.md)。

## 9. 设计边界

- Training Core 是唯一可直接访问业务数据库的模块。
- Trainer MCP、Web 和未来 Anki Bridge 只能通过 Core API 获取权威数据。
- 调度行为必须位于 Scheduler Provider 合同之后。
- Anki 是可选插件和投影，不是 MindTrain 权威题库。
- 不为当前单用户 MVP 提前拆分微服务。

产品目标与架构说明见[目标需求](目标需求.md)和[概要设计](概要设计.md)。
